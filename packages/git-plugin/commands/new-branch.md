# 创建分支

根据用户提供的名字，**侦测项目现有分支用的命名规范**，按该规范把名字转成合规的英文分支名，再 `git checkout -b` 创建并切换。**只在用户明确触发时执行**。

用户提供的名字来自 `$ARGUMENTS`。如果为空，让用户补充一个名字，不要瞎猜。

## 执行流程

### 1. 并行收集上下文
同时跑（都是只读）：
- `git branch -a`（本地 + 远程所有分支，用于侦测命名规范）
- `git branch --show-current`（当前分支，作为新分支的 base）
- `git status`（确认工作区状态）

### 2. 侦测命名规范（convention detection）
扫描 `git branch -a` 列出的所有分支名（忽略 `HEAD ->`、`remotes/origin/` 前缀只看分支主体），归纳项目主流用哪种规范。常见几类：

- **类型前缀 + 斜杠**：`feat/xxx`、`feature/xxx`、`fix/xxx`、`bugfix/xxx`、`hotfix/xxx`、`chore/xxx`、`release/xxx`、`docs/xxx`（最主流，Git Flow / GitHub Flow 风格）
- **类型前缀 + 短横线**：`feature-xxx`、`fix-xxx`
- **带 issue 号**：`feat/123-xxx`、`PROJ-123-xxx`（Jira 风格）
- **无前缀纯短横线小写**：`add-login`、`update-readme`
- **其它**：如下划线 `feat_xxx`

判定方法：
- 统计各分支的前缀模式、分隔符（`/` vs `-` vs `_`）、大小写。
- 取出现次数最多的一种作为项目规范。
- 若分支太少、只有 `main`/`master`/`dev`/`develop` 等基础分支无法判断，**回退到默认规范** `<type>/<name>`（斜杠前缀），type 默认 `feat`。
- 一句话把侦测结论告诉用户（例如「项目用 `feat/xxx` 斜杠前缀规范」），保证最终命名可解释。

### 3. 把名字转成英文 slug（核心）
用户给的"名字"是分支的语义部分（描述这个分支干嘛）。按下列规则转换：

- **英文输入**：规范化即可——转小写，空格和标点换成分隔符（斜杠规范里语义段内部用 `-`）。例：`Add user login` → `add-user-login`。
- **中文输入，优先意译**：能简洁意译成常用英文就意译，控制在 1–3 个词。例：`用户登录` → `user-login`、`修复支付回调` → `fix-payment-callback`、`重构订单模块` → `refactor-order-module`。
- **专有名词 / 产品名 / 人名 / 无通用译法 → 用拼音**：例：人名 `张三` → `zhangsan`、产品名 `飞鸽` → `feige`。
- **按长度决定是否用拼音首字母**：
  - 意译或拼音后如果 slug 太长（启发式阈值：拼音全拼超过约 12 个字母，或中文超过 4 个字），改用**拼音首字母缩写**，避免分支名又长又难读。例：`商品库存预警` 全拼太长 → 首字母 `spkcyj`。
  - 这是启发式（heuristic）判断，不是硬规则，结合可读性灵活把握。目标是"**短，且还能认出大概意思**"。
  - 用了拼音或首字母时，在最后输出里附上中文原意，方便日后理解。

### 4. 推断类型前缀（type）
如果项目规范带类型前缀：
- 用户可在 `$ARGUMENTS` 里显式指定类型，如 `fix 登录崩溃` → type 取 `fix`。
- 否则从名字语义推断：修复 / bug / 崩溃 → `fix`；文档 → `docs`；重构 → `refactor`；构建 / 依赖 / 配置 → `chore`；其余新功能 → `feat`（默认）。
- 类型用项目里实际出现的形式（项目用 `feature/` 就别写成 `feat/`）。

### 5. 组合并校验分支名
- 组合：`<前缀><分隔符><slug>`，得到最终分支名。
- 校验合法性，不满足就调整：
  - 不含空格；
  - 不含 git 非法字符 `~ ^ : ? * [ \ ` 和空白；
  - 不出现连续 `//`、不以 `/` 或 `.` 结尾、不以 `.lock` 结尾、不以 `/` 开头；
  - 默认全小写（除非项目规范明显用大写 issue key 如 `PROJ-123`）。
- 若与现有分支重名，提示用户并加区分后缀或请用户改名，不要覆盖已有分支。

### 6. 执行创建
- 用 `git checkout -b <最终分支名>` 创建并切换。
- 若 `git status` 显示工作区有未提交改动，**先提醒用户**这些改动会一起带到新分支（不自动 stash、不自动 commit），再创建。

### 7. 输出
创建成功后，用几句话告诉用户：
- 侦测到的命名规范（一句话）
- 原始名字 → 最终分支名 的转换说明（用了拼音 / 首字母时附中文原意）
- 已切换到新分支，并跑一次 `git status` / `git branch --show-current` 展示当前状态
- 是否要推送（不要主动推，让用户决定）

## 不要做的事

- 名字为空或含义不清时先问用户，不要硬造分支名
- 不要 `git push` / `git push -u`，除非用户明确要求
- 不要修改 git config
- 工作区有改动时不自动 stash / commit，只提醒
- 不要覆盖或删除已存在的同名分支
- 不要把 `claude` / `claude code` 等字样放进分支名（用户没要求时）
